OWASP MASVS explained for mobile teams

Compliance · April 5, 2024 · 7 min read

What is MASVS?

The OWASP Mobile Application Security Verification Standard (MASVS) is the industry-standard framework for mobile app security requirements. Developed by the Open Web Application Security Project, MASVS defines a baseline of security controls that every mobile app should implement, along with enhanced requirements for apps that handle sensitive data.

MASVS is not a penetration testing methodology — it's a requirements framework. It tells you what security controls your app needs, organized into clear categories: data storage, cryptography, authentication, network communication, platform interaction, code quality, and resilience against reverse engineering.

L1 vs L2 controls

MASVS defines two primary verification levels:

  • MASVS-L1 (Standard Security): The baseline that every mobile app should meet. Covers fundamental security hygiene — proper data storage, network security, input validation, and basic platform security. Appropriate for apps that don't handle highly sensitive data.
  • MASVS-L2 (Defense-in-Depth): Enhanced requirements for apps that handle sensitive data — banking, healthcare, government, and payment applications. Includes all L1 controls plus advanced protections: certificate pinning, anti-tampering, root detection, code obfuscation, and runtime integrity verification.

Additionally, MASVS defines a Resilience category (MASVS-R) for apps that need protection against reverse engineering and client-side attacks. This includes requirements for obfuscation, anti-debugging, device integrity verification, and runtime instrumentation detection.

How to achieve compliance

Achieving MASVS compliance requires a systematic approach:

  • Gap analysis: Map your current security controls against MASVS requirements to identify what's missing. The OWASP MSTG (Mobile Security Testing Guide) provides specific test cases for each requirement.
  • Prioritize by risk: Not all requirements carry equal weight. Focus first on data storage (where most breaches originate), network security (MITM prevention), and authentication controls.
  • Implement in layers: Start with L1 controls in your development process, then add L2/R controls through runtime protection tooling. Trying to implement everything manually is prohibitively expensive.
  • Continuous verification: MASVS compliance isn't a one-time achievement. Automated testing in CI/CD ensures that new releases maintain compliance as code changes.
